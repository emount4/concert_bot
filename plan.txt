Plan: Concert Reviews Telegram Mini App (MVP → Production)

Цель
Сделать Telegram Mini App + Go (Gin) backend + React frontend + Postgres так, чтобы:
- MVP был достижим и закрывал весь основной UX.
- Проект был “серьёзным”: не ложился в проде из‑за отсутствия таймаутов/миграций/наблюдаемости, и не запирался в Telegram-экосистеме.

Ключевые решения
- Авторизация: Telegram WebApp initData (автоматическая) с серверной валидацией подписи.
- Пользовательская модель: отдельная сущность users + таблица привязок user_identities(provider=telegram).
- Модерация: статусы рецензии pending/approved/rejected.
- Рейтинг: 4 объективных параметра (1–10) + vibe (1–10) → vibeMultiplier 1..1.6072 → общий балл capped до 90.
- Медиа в MVP: хранить URL в БД (самый практичный старт), но проектировать так, чтобы позже легко переехать на S3/R2 (без перелома схемы).

============================================================
Раздел 1 — Зафиксированное MVP (продукт)
============================================================

1) Авторизация
- Пользователь открывает мини-апп → происходит автоматическая авторизация (без форм).
- Авторизация реализуется через WebApp initData:
	- фронт получает initData
	- отправляет на backend
	- backend валидирует подпись и создаёт/находит пользователя
	- backend создаёт сессию

2) Профиль
- Отображается:
	- дата регистрации
	- описание (bio)
	- количество рецензий
	- список твоих рецензий (желательно со статусом модерации)

3) Главная страница
- Секция “последние рецензии” (горизонтальный скролл).
- Секция “последние концерты” (горизонтальный скролл или сетка с горизонтальным скроллом).
- При клике на концерт → открывается страница концерта.

4) Страница концерта
- Отображается:
	- информация о концерте (артисты, площадка, дата)
	- средняя оценка (только approved рецензии)
	- количество рецензий (только approved)
	- список рецензий
- Если пользователь не писал рецензию — показывается форма создания.

5) Рецензия
- Текст: 300..8500 символов (валидировать на сервере).
- Оценка по 5 параметрам (описано в Разделе 2).

6) Админ
- Новые концерты/артистов/площадки добавляет администратор.
- Администратор модерирует рецензии.
- Администратор может банить пользователей.
- Это реализовано через отдельную админ-панель (UI) и admin API.
- Нужны все CRUD операции.

7) Сущности в БД
- Artists, Venues, Concerts — отдельные сущности.
- Концерт может иметь несколько артистов (many-to-many).

============================================================
Раздел 2 — Рейтинг (точная спецификация)
============================================================

Объективные параметры (1–10 каждый):
1) Исполнение
2) Динамика / трек‑лист
3) Подача артиста, взаимодействие с аудиторией
4) Работа звукаря (продакшн)

Субъективный параметр:
5) Вайб (1–10)

Вайб → множитель:
- vibeMultiplier ∈ [1, 1.6072]
- Нужна детерминированная функция vibeMultiplier(vibe) (одинаковая на фронте и бэке).
	Практично сделать её строго в backend (источник истины), а на фронте либо:
	- повторить ту же функцию 1-в-1 для превью, либо
	- запрашивать “превью overall” с сервера при изменении слайдера (можно позже).

Итоговая формула:
- objectiveSum = sum(4 объективных)
- overallRaw = objectiveSum * 1.4 * vibeMultiplier
- overallScore = min(90, overallRaw)

Рекомендации хранения:
- В БД хранить:
	- все 4 объективных + vibe (как int)
	- vibe_multiplier (numeric)
	- overall_score (numeric)
- overall_score пересчитывать на сервере при create/update рецензии.

============================================================
Фаза 1 — Production foundation backend (1–2 дня)
============================================================

Цель: сервис не падает от базовых прод-проблем (таймауты, shutdown, логирование, readiness).

1.1 Конфиг (env)
Добавить в конфиг (и валидировать где нужно):
- PORT
- LOG_LEVEL
- DATABASE_URL
- TELEGRAM_BOT_TOKEN
- ADMIN_TELEGRAM_IDS (csv)
- SHUTDOWN_TIMEOUT (duration, напр. 10s)
- HTTP_READ_HEADER_TIMEOUT (duration, напр. 10s)
- HTTP_READ_TIMEOUT / HTTP_WRITE_TIMEOUT / HTTP_IDLE_TIMEOUT (рекомендуется для прод)
- SESSION_SECRET (если cookie-сессии) или JWT_SECRET (если JWT)

1.2 Логи и request_id
- Оставить единый дефолтный slog (SetDefault) в main.
- Добавить middleware request_id:
	- брать X-Request-ID если пришёл
	- иначе генерировать
	- добавлять в ответ
- Расширить access-logs:
	- request_id
	- user_agent
	- ошибки из gin.Context.Errors (минимум количество/первая)

1.3 Health/Readiness
- /health: просто “жив”
- /ready: проверка БД (ping) и базовой готовности

1.4 Graceful shutdown
- Таймауты сервера и shutdown брать из конфига.
- На SIGTERM/CTRL+C корректно закрывать сервер.

============================================================
Фаза 3 — Telegram Mini App Auth (1–2 дня)
============================================================

3.1 Endpoint авторизации
POST /api/auth/telegram
- input: initData
- сервер валидирует подпись initData
- upsert user + identity
- создаёт сессию:
	- рекомендовано: httpOnly cookie (если один origin)
	- альтернативно: JWT (быстрее, но чаще хуже по безопасности)

3.2 Middleware
- auth middleware: достаёт user_id
- ban middleware: запрещает write-операции при активном бане
- admin middleware: проверяет пользователя по ADMIN_TELEGRAM_IDS

Фаза 4 — Public API под Mini App (3–6 дней)
============================================================

4.1 Профиль
GET /api/me
- created_at, display_name, bio, reviews_count

GET /api/me/reviews?status=all|approved|pending|rejected
- список рецензий пользователя (+ статус)

4.2 Главная
GET /api/feed
- latest_reviews (approved, limit)
- latest_concerts (limit)

4.3 Концерты
GET /api/concerts/:id
- concert + venue + artists
- avg_overall_score (только approved)
- reviews_count (только approved)
- has_my_review + my_review (если есть)

GET /api/concerts/:id/reviews?limit&offset
- только approved

4.4 Рецензии
POST /api/concerts/:id/reviews
- валидировать текст 300..8500
- валидировать рейтинги 1..10
- рассчитать vibeMultiplier и overallScore по формуле
- статус pending

PUT /api/reviews/:id
- только владелец
- после изменения: pending (рекомендация)

4.5 Агрегации
- считать avg и count по status=approved.
- сначала SQL агрегации, оптимизация позже.

Фаза 5 — Admin API + Admin Panel (5–10 дней)
============================================================

5.1 Admin API (под /api/admin)
Artists CRUD:
- GET/POST/PUT/DELETE /api/admin/artists

Venues CRUD:
- GET/POST/PUT/DELETE /api/admin/venues

Concerts CRUD:
- GET/POST/PUT/DELETE /api/admin/concerts
- управление артистами концерта:

- GET /api/admin/reviews?status=pending
- POST /api/admin/reviews/:id/approve

Users ban/unban:
- GET /api/admin/users?query=
- POST /api/admin/users/:id/ban (+ until, reason)
- POST /api/admin/users/:id/unban

5.2 Admin UI

- страницы:
	- artists CRUD
