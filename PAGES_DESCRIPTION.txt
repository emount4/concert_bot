CONCERT_BOT — описание страниц (frontend)

Скоуп документа
- Описывается пользовательский интерфейс фронтенда (React + TypeScript + Vite) и то, как он реально устроен в текущем коде.
- Описания опираются на роутинг из front/src/App.tsx и компоненты страниц в front/src/pages/*.
- Данные: либо mock-режим, либо API-режим (переключается переменной VITE_DATA_SOURCE).


1) Общие принципы приложения

1.1. Роутинг и защита страниц
- Роутинг: React Router (BrowserRouter).
- Главный роутер: front/src/App.tsx.
- Почти все страницы обёрнуты в GuardedRoute.
  - Если пользователь не аутентифицирован, происходит редирект на /login.
  - Состояние редиректа сохраняется в location.state.from.pathname (куда пользователь пытался попасть).

1.2. Режимы данных (Mock/API)
- Источник данных выбирается через VITE_DATA_SOURCE:
  - mock (по умолчанию) — данные берутся из front/src/data/mock*.ts.
  - api — данные берутся с бэкенда через apiRequest, базовый URL: VITE_API_BASE_URL (по умолчанию http://127.0.0.1:8080/api/v1).
- Центральная точка загрузки данных: front/src/api/AppDataProvider.tsx.
  - Внутри вызывает loadAppBootstrapData() из front/src/api/repository.ts.
  - Провайдит data/isLoading/error + refresh() через контекст.

1.3. Общий layout
- На большинстве страниц (кроме /login и /register) показываются:
  - Левая вертикальная навигация (TopBar) с иконками.
  - Верхний Header (профильное меню + кнопка “Предложить концерт”).
  - Footer внизу (скрывается на /login).
- На /login и /register применяется authMode: скрываются TopBar/Header, другой визуальный контейнер.

1.4. Навигация по разделам
- TopBar (левая колонка) содержит только контентные разделы:
  - /home, /concerts, /reviews, /artists, /venues, /faq, /about
- Header содержит меню профиля (dropdown):
  - “Мой профиль” → /profile/<myUsername> (или /profile если username не вычислен)
  - “Настройки” → /settings
  - “Админ‑панель” → /admin (показывается только если у пользователя есть права)
  - “Выйти” → logout + redirect на /login

1.5. Статические HTML-документы (не React-страницы)
- Это файлы в front/public, открываются в новой вкладке браузера:
  - /terms.html
  - /privacy.html
  - /moderation.html
- Ссылки на них находятся в Footer и в Settings.

3) Описание страниц

3.1. Страница входа
URL: /login
Компонент: front/src/pages/LoginPage.tsx
Доступ: публичная (без GuardedRoute). Если уже logged in — редирект на /home.
Цель: вход по email/паролю (мок).
UI:
- Карточка с заголовком “Вход” и подзаголовком.
- Поля: Email, Пароль.
- Ошибка отображается текстом под полями.
- Кнопка “Войти” меняет текст на “Проверяем...” во время запроса.
- Ссылка на регистрацию.
Логика:
- При submit вызывается loginWithEmailPassword() (mock).
- После успешного входа: navigate(redirectTo).
  - redirectTo берётся из location.state.from.pathname или по умолчанию “/concerts”.
Особенности:
- Подсказка мок-доступа: demo@concert.bot / demo123.


3.2. Страница регистрации
URL: /register
Компонент: front/src/pages/RegisterPage.tsx
Доступ: публичная. Если уже logged in — редирект на /home.
Цель: регистрация + подтверждение email кодом (мок).
UI:
- Step 1 (register): Имя, Email, Пароль, Повтор пароля.
- Step 2 (verify): поле для кода (6 цифр).
- Общая карточка, ошибки под полями.
Логика:
- Step register:
  - Проверка совпадения паролей.
  - registerWithEmail() (mock) → при успехе переключение на step verify.
- Step verify:
  - verifyEmailCode() (mock) → при успехе редирект на /concerts.
- В verify-step показывается подсказка мок-кода: getMockVerificationCodeHint().


3.3. Главная
URL: /home
Компонент: front/src/pages/HomePage.tsx
Доступ: GuardedRoute (нужна авторизация).
Цель: “витрина” проекта: топы концертов, лидеры по параметрам, свежие рецензии, глобальная статистика.
Данные:
- Используется @tanstack/react-query.
- Топы/лента/статистика берутся из front/src/mocks/home (fetch* функции).
UI и блоки:
A) “Лучшие по оценке”
- Горизонтальная карусель Top-5.
- Каждый элемент ведёт на /concerts/<id>/rate.
- Поверх карточки — крупный бейдж “место в топе” (1..5).
- Есть “Смотреть все” → /concerts.

B) “Самые популярные”
- Аналогично Top-5 карусель.
- Также ведёт на /concerts/<id>/rate.

C) “Лидеры по параметрам”
- Три колонки: Артисты / Площадки / Концерты.
- В каждой колонке выпадающий select параметра оценки (ScoreKey).
- Ниже — список из 5 строк (RankBadge + название + значение /10 + количество).
- Каждая строка кликабельна и ведёт на href из моков (обычно на /artists?artistId=... или /venues?venue_id=... или /concerts/:id/rate — зависит от моков).

D) “Свежие рецензии”
- Подгружается лениво: только когда якорь секции попадает в viewport (IntersectionObserver).
- Пока секция не в зоне видимости — подсказка “Прокрутите ниже, чтобы загрузить ленту”.
- После загрузки — grid карточек ReviewCard.
- Ссылка “Смотреть все” → /reviews.

E) “Статистика”
- Также ленивый запрос по IntersectionObserver.
- Рендерится как единая горизонтальная панель stat-item (users/concerts/artists/venues/reviews).

Состояния:
- Loading/ошибка на каждый блок.
- Для top-секций есть SectionErrorBoundary с fallback.


3.4. Список концертов
URL: /concerts
Компонент: front/src/pages/ConcertsPage.tsx
Доступ: GuardedRoute.
Цель: поиск, фильтрация, сортировка и пагинация концертов.
Данные:
- Из useAppData(): data.concerts.
UI:
- Заголовок “Концерты”.
- Панель controls:
  - Search: поиск по концерту/артисту/площадке.
  - City select (все города / конкретный город) — формируется из концертов.
  - Sort select: дата/оценка/число рецензий/название.
  - Кнопка направления сортировки (↑/↓).
  - “Сбросить фильтры”.
- Grid карточек концертов.
  - Каждая карточка — ссылка на /concerts/<id>/rate.
- Пагинация:
  - Размер страницы: 12.
  - Кнопки страниц + многоточия, при смене страницы выполняется scrollToTop().
Логика фильтрации:
- City filter сравнивает concert.venue.city.
- onlyRated: оставляет только концерты, где avg_rating_total != null.
- upcomingOnly: оставляет только концерты с датой >= now.
- Поиск: по title, venue.name/city, artist names.
Состояния:
- Если данных нет по фильтрам — “По выбранным фильтрам концертов не найдено”.


3.5. Детальная страница концерта + оценивание/рецензия
URL: /concerts/:concertId/rate
Компонент: front/src/pages/RateConcertPage.tsx
Доступ: GuardedRoute.
Цель:
- Просмотр “шапки” концерта (постер, дата, площадка, артисты, средняя оценка).
- Заполнение оценки по 90-балльной системе и текста рецензии.
- Просмотр списка рецензий по этому концерту.
Данные:
- Из useAppData(): concerts + reviews.
- concert определяется по concertId из params.
UI: верхний hero
- Постер (img или fallback).
- Теги:
  - Дата (формат ru-RU).
  - Площадка (кликабельная ссылка на /venues?venue_id=<id>).
- Кнопка “в избранное” (сердечко) — локальный toggle state, без синхронизации с профилем.
- Название концерта.
- Чипы артистов: ссылки на /artists?artistId=<id>.
- Средняя оценка концерта (если есть) в виде RatingBreakdownBadge с breakdown по параметрам.

UI: форма оценки
- 4 объективных ползунка 1..10:
  - Исполнение
  - Динамика / трек-лист
  - Харизма
  - Звук
- 1 субъективный ползунок “Вайб” 1..10 (отдельный блок шире).
- Поле “Название рецензии”.
- Textarea “Текст рецензии” с подсчётом символов (0..8500).
- Utility ряд:
  - “Очистить черновик” → открывает confirm modal.
  - “Критерии 90-балльной системы оценивания” → открывает modal с описанием параметров и формулы.
  - “📎 Прикрепить медиа (N)” → modal прикрепления.
- Блок submit:
  - Отображает рассчитанный итог (overallRounded) /90.
  - Кнопка ✓ (в текущем коде — UI-кнопка без реальной отправки).

Логика расчёта итоговой оценки
- Объективная сумма: performance + setlist + crowd + sound.
- Вайб используется как множитель: 1.6072^((vibe - 1)/9).
- Итог: objectiveSum × 1.4 × vibeMultiplier, ограничение max 90.

Черновик (localStorage)
- Ключ: draft_<concertId>.
- При заходе на страницу пытается загрузить JSON черновика.
- Автосохранение с debounce 1200ms; появляется toast “Черновик сохранен”, который скрывается через 1800ms.
- “Очистить черновик” сбрасывает поля и удаляет localStorage.

Модальные окна
- Критерии: объяснение параметров и формулы.
- Подтверждение очистки: “Удалить черновик?”
- Прикрепление медиа:
  - input file accept image/*,video/*, multiple.
  - список выбранных файлов с размером (форматирование Б/КБ/МБ) и кнопкой “Удалить”.
  - кнопка “Очистить список” удаляет все выбранные файлы.

Блок рецензий концерта
- Заголовок “Написанные рецензии” + бейдж количества.
- Пагинация (по 10 на страницу).
- ReviewCard показывается в expanded mode.

Состояния:
- Loading/ошибка/“концерт не найден”.

3.7. Артисты — список + детальная
URL: /artists
Компонент: front/src/pages/ArtistsPage.tsx
Доступ: GuardedRoute.
Как устроена “детальность”:
- Детальная страница реализована в том же компоненте.
- Выбор артиста определяется query param: artistId.
  - Пример: /artists?artistId=12
- Если artistId отсутствует или не найден — показывается список.

A) Список артистов
Цель: поиск/фильтрация/сортировка/пагинация.
Данные:
- artists, concerts, reviews из useAppData().
- artistStats вычисляет по каждому артисту:
  - concertsCount
  - reviews_count (через связь концерт→артист + число отзывов по концертам)
UI controls:
- Search по имени артиста.
- Filter по наличию рецензий: любые / есть / нет.
- Sort: по оценке / по алфавиту.
- Направление сортировки (↑/↓).
- “Сбросить фильтры”.
- Grid карточек ArtistCard.
- Пагинация: 12 на страницу.

B) Детальная карточка артиста
Цель: показать шапку артиста, средние оценки, последние концерты и рецензии.
UI:
- Верхняя строка: “Артист” + ссылка “Все артисты” (back).
- Hero:
  - Фото (img) или пусто.
  - Название + сердечко “в избранное” (локальный toggle, сбрасывается при смене selectedArtist).
  - Статистика: концертов, рецензий.
  - Средняя оценка (RatingBreakdownBadge) + breakdown по параметрам (до десятых) на основе отзывов.
- Секции:
  - “Последние концерты”: горизонтальный scroll, карточки ConcertCard, ведут на /concerts/:id/rate.
  - “Последние рецензии”: горизонтальный scroll, ReviewCard.
Состояния empty:
- Если нет концертов/рецензий — соответствующие placeholder.

3.9. Профиль (мой)
URL: /profile
URL: /profile/:username (в текущем роутере рендерит тот же компонент, игнорирует :username)
Компонент: front/src/pages/ProfilePage.tsx → ProfileScreen(kind="me")
Доступ: GuardedRoute.
Цель: единый экран профиля с табами “Рецензии / Избранное / Лайкнутое”.
Данные:
- База берётся из useAppData() (data.profile + data.reviews + каталоги).
- Социальные моки (избранное/лайки) берутся из front/src/data/mockUsers.ts:
  - MOCK_FAVORITES_BY_USERNAME
  - MOCK_LIKED_REVIEW_IDS_BY_USERNAME
- Для имитации “сетевой задержки” используется sleep(350).

UI layout
- Две колонки:
  - Sidebar:
    - Identity card (аватар/фоллбек, @username, бейдж “Админ” если включён dev-admin, дата регистрации, bio).
    - Card “Статистика” списком строк: Рецензий / Лайков получено / Лайков поставлено.
  - Main:
    - Баннер профиля (img если есть).
    - Tabs.
    - Content выбранного таба.

Поведение при удалённом аккаунте
- Если is_active = false:
  - аватар/баннер скрываются, показывается “Аккаунт удален”, bio не показывается.
  - в рецензиях автор показывается как “Удаленный пользователь”.

Таб: “Рецензии”
- Список ReviewCard в колонку.
- Для своего профиля:
  - для рецензий со статусом pending/rejected в ReviewCard прокидывается проп moderation.
  - В карточке появляется кнопка “Статус модерации” и модалка с деталями (внутри ReviewCard).
- Для чужого профиля:
  - видны только approved.

Таб: “Избранное”
- Вертикально 3 блока: Концерты / Артисты / Площадки.
- В каждом блоке:
  - Заголовок + декоративная линия.
  - Ряд из максимум 5 элементов в виде кругов-иконок с подписью.
  - Каждый элемент — Link на соответствующую детальную страницу:
    - концерт → /concerts/:id/rate
    - артист → /artists?artistId=:id
    - площадка → /venues?venue_id=:id
- Если в блоке 0 элементов — внутри ряда показывается текст “нет избранных” (при этом ряд сохраняет выделенное пространство).

Таб: “Лайкнутое”
- Список ReviewCard.
- Из лайкнутого исключаются рецензии самого пользователя.
- Если пусто — “Нет лайкнутых рецензий”.

Скелетоны
- Во время загрузки показываются skeleton-блоки (ProfileSkeleton).

3.11. Настройки
URL: /settings
Компонент: front/src/pages/SettingsPage.tsx
Доступ: GuardedRoute.
Цель: страница настроек из 5 смысловых блоков (многое — мок, но UI и часть логики реализованы).
Данные:
- useAppData().data.profile (как базовое состояние).
- Админ-очередь изменений профиля хранится в front/src/data/adminStore (localStorage-like).

Блок 1: “Публичный профиль (модерируемый)”
- Username:
  - input, кнопка “Проверить” (мок-проверка уникальности), статусы available/taken/invalid.
  - “Отправить на модерацию” — создаёт заявку enqueueProfileChangeRequest(type=username).
- Bio:
  - textarea + “Отправить на модерацию” → enqueueProfileChangeRequest(type=bio).
- Аватар:
  - выбор файла + превью (текущая/новая).
  - “Загрузить (на модерацию)” → enqueueProfileChangeRequest(type=avatar) с dataUrl.
- Баннер:
  - выбор файла + превью.
  - “Загрузить (на модерацию)” → enqueueProfileChangeRequest(type=banner) с dataUrl.
- “Статус модерации”:
  - кнопка “Открыть” → модалка со списком заявок и их статусом.

Блок 2: “Аккаунт и безопасность”
- Email readonly.
- “Смена пароля” (mock): форма old/new/repeat, результат выводится сообщением.
- “Привязка Telegram” (mock): кнопка привязать/отвязать, отображает @mock_user.
- “Выход”: logout + redirect на /login.

Блок 3: “Настройки интерфейса (UX)”

3.14. Админ‑панель
URL: /admin
Компонент: front/src/pages/AdminPage.tsx
Доступ: GuardedRoute + доп. проверка isAdmin.
Как определяется доступ:
- В App.tsx isAdmin = resolveIsAdmin() (dev-флаг) ИЛИ роль current account из data.admin.accounts (admin/super-admin).
- Если isAdmin=false — показывается экран “Доступ только для администратора” и кнопки включить/сбросить dev-админа (с reload).

Структура админки
- Заголовок + intro + бейдж Pending.
- Tabs (верхний tablist):
  - Модерация
  - Очередь
  - Артисты
  - Площадки
  - Города
  - Концерты
  - Аккаунты
  - Логи (только для super-admin)

Таб “Модерация” (рецензии)
- SubTabs потоков: Новые(pending) / Одобренные / Отклонённые.
- Карточка рецензии: концерт, статус, автор (ссылка на /users/:username), дата, rating_total, текст.
- Действия:
  - “Медиа (N)” → modal просмотра вложений по одному (←/→, счётчик).
  - “Одобрить” → открывает modal “Одобрение с правками” (можно править текст и снять галочки у медиа).
  - “Отклонить” → status=rejected.
  - “Вернуть в pending”.

Таб “Очередь”
- SubTabs:
  - Модерация профилей (заявки на username/bio/avatar/banner)
  - Предложения (концерты)
- Профили:
  - Карточки заявок со статусом и предпросмотром diff.
  - pending → кнопки “Принять/Отклонить”.
  - При approve обновляется profileStore override и вызывается refreshAppData().
- Предложения:
  - Карточки предложений.
  - “Создать на основе” переводит suggestion в created и переносит на таб “Концерты” с предзаполненной формой.

Табы “Артисты / Площадки / Города / Концерты”
- Типовой паттерн: слева форма создания/редактирования, справа список с поиском.
- Для медиа (фото/афиши) используется onMediaPick → dataUrl.
- Есть базовые действия: сохранить/очистить/изменить/удалить.
- Для концерта:
  - выбор площадки через модалку (поиск).
  - выбор артистов через модалку (мультивыбор).

Таб “Аккаунты”
- Поиск.
- Показывает роль, бан, “текущий аккаунт”.
- Действия:
  - бан/разбан (не для текущего аккаунта).
  - назначение админа / понижение (только super-admin).

Таб “Логи”
- Только для super-admin.
- Список audit log: сообщение + дата + кто сделал (displayName + роль).
